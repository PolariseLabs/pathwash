# Product

## Register

brand

## Users

Engineers who accept file or archive uploads and have been burned by what arrives: Windows-built zips, macOS junk entries, unicode mismatches, hostile paths. They land on the README from npm or GitHub search, usually mid-bug, and decide within a minute whether this library solves their problem.

## Product Purpose

pathwash is a zero-dependency TypeScript library that turns dirty archive and upload paths into a canonical form, with validation defined as a fixed point of normalisation so checks and transforms cannot drift apart. It exists because this class of bug gets re-fixed ad hoc in every codebase that handles uploads. Success: a reader can tell within the first screen of the README exactly which failure modes are covered, and adopt it with one import.

## Brand Personality

Precise, dry, engineer-to-engineer. Reads like a maintainer's notes, not a pitch. Three words: terse, concrete, trustworthy.

## Anti-references

- AI-sounding copy: aphoristic cadence ("X, not Y" slogans as recurring rhythm), em dashes, triadic flourishes, "The real win is" framing.
- Marketing README style: hero taglines, buzzwords (seamless, powerful, blazing-fast), feature checklists with emoji.
- Over-documented style: restating what the code example already shows.

## Design Principles

- Show the failure mode before the feature. Every capability is justified by a concrete bug.
- Code examples are the primary prose. Text around them earns its place or goes.
- One screen to a decision. Install, one usage block, and the coverage table come first.
- British spelling everywhere, including API identifiers.

## Accessibility & Inclusion

Docs only (no UI). Code blocks must be copy-pasteable and correct. Tables need header rows. Link text carries standalone meaning.
